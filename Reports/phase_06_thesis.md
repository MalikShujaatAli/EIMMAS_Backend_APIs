# Phase 6 Thesis: First Unified API (The Monolith)

## Strategic Intent
Phase 6 was the pivotal transition from desktop-only scripts to network-accessible APIs. The driving question was: "How do we expose these emotion models to a Flutter mobile app over HTTP?" The answer was to combine face and voice models into a single FastAPI application with three POST endpoints: `/predict/image`, `/predict/video`, and `/predict/voice`. A secondary goal was to package this monolith as a standalone Windows executable using PyInstaller (`phase06_fusion_api_pyinstaller_spec.spec`).

## Scope & Boundaries
Two script variants were produced. `phase06_fusion_api_monolith.py` (238 lines) loaded the Phase 1 CNN (`face_emotion_model.h5`, 48×48) and a TF 2.12-compatible voice LSTM (`final_lstm_model_tf212.h5`), serving them both from port 8000. `phase06_fusion_api_old_endpoints.txt` was a parallel variant referencing different model files (`facial_emotion.json` + `.h5`, `speech_emotion_model_7(2).h5`) with slight calibration differences. No text endpoint existed in either variant. The `phase06_fusion_api_pyinstaller_spec.spec` file attempted to bundle the entire API into a single `.exe` using PyInstaller, embedding model weights as binary data.

## Failure Analysis
Phase 6 failed on five fronts. (1) **Accuracy**: The Phase 1 CNN (48×48, 57% accuracy) was still the face model; it had not been retrained. (2) **Concurrency**: All `async def` endpoints ran synchronous `model.predict()` and `librosa.load()`, blocking the event loop and freezing the server for all users during any single inference. (3) **False positives**: Haar Cascades detected non-human objects as faces. (4) **Sample rate mismatch**: The voice endpoint used `librosa.load(audio_stream, sr=22050)` despite the voice model being trained on 16kHz audio, causing feature distribution drift. (5) **Scaling impossibility**: Both models shared a single process, single thread, single worker. The `reload=True` flag in `uvicorn.run()` caused the development server to reload on file changes, which in production meant random restarts and memory leaks. The PyInstaller bundling attempt (`phase06_fusion_api_pyinstaller_spec.spec`) was ultimately abandoned because TensorFlow's dynamic library loading was incompatible with PyInstaller's static bundling approach.
