# Phase 7 Thesis: Second Attempt — Separated APIs (Still Flawed)

## Strategic Intent
Phase 7 was the first microservices attempt — splitting the monolith into three dedicated FastAPI servers (Audio on 8002, Video on 8002, Text on 8001). Each service owned its own model, its own preprocessing, and its own endpoint. The hypothesis was that separation would fix the concurrency and scaling problems. Phase 7 also introduced several critical upgrades: MediaPipe replaced Haar Cascades, CLAHE was introduced for lighting correction, the retrained FERPlus model (`fer_best_model.keras`, 112×112, 81%) replaced the Phase 1 CNN, the text API gained batch inference and a proper `AttentionLayer`, and the audio API gained silence trimming, file size limits, and CORS middleware.

## Scope & Boundaries
Three scripts were developed: `2nd attempt Audio.txt` (202 lines) added `librosa.effects.trim`, `CONFIDENCE_THRESHOLD`, `MAX_FILE_SIZE_MB`, extension whitelisting, and `finally` block cleanup for temp files. `2nd attempt Video.txt` (242 lines) introduced MediaPipe `FaceDetection`, CLAHE, 112×112 preprocessing, and `cv2.INTER_CUBIC` interpolation. `2nd attempt Text.txt` (162 lines) introduced the Keras-serializable `AttentionLayer` with `K.tanh`/`K.dot` ops, batch inference via `model.predict(padded_batch)`, and confidence thresholding per sentence.

## Failure Analysis
Despite the structural improvements, Phase 7 retained three critical flaws. (1) **Event loop blocking**: All three scripts defined endpoints as `async def` but executed synchronous operations (`librosa.load`, `model.predict`, `cv2.VideoCapture`) on the main event loop thread. This meant a single user's request still froze the server for everyone. `asyncio.to_thread()` was not introduced until Phase 9. (2) **Video frame sampling**: The video endpoint used `frame_id % 10` (every 10th frame) regardless of source FPS, and called `model.predict()` per-frame without batching. (3) **NLTK runtime download**: The text API called `nltk.download('punkt', quiet=True)` at global scope during server boot, creating a hard dependency on internet connectivity and NLTK server availability.
